# V0.4 Runtime Observability Acceptance

V0.4 adds a read-only runtime observability and introspection control surface.
It lets a user or agent inspect the current Session, Task, queue/lease,
EventLog timeline, approval/commit handoff, worker lane/diff evidence, and
static runtime architecture reports without running workers or mutating state.

## Environment

- Rust toolchain for workspace edition 2024.
- `cargo` and `git` on `PATH`.
- A reachable PostgreSQL database.
- `HARNESS_DATABASE_URL` set to that database, or pass `--database-url` on each
  CLI command.
- For automated PostgreSQL integration tests, set `HARNESS_TEST_DATABASE_URL`.

For Docker, existing PostgreSQL, or WSL/k3s setup, see
`docs/development/postgresql.md`.

## Command Surface

All commands have human-readable `key=value` output by default. Commands marked
with `--json` also emit structured JSON intended for agents and other tooling.

| Area | Human-readable command | Structured JSON |
| --- | --- | --- |
| Session | `harness-cli session inspect <SESSION_ID>` | `--json` |
| Task | `harness-cli session task inspect <SESSION_ID> <TASK_ID>` | `--json` |
| Queue and lease | `harness-cli session task queue-inspect --session-id <SESSION_ID>` | `--json` |
| EventLog timeline | `harness-cli session timeline <SESSION_ID> [--task-id <TASK_ID>]` | `--json` |
| Approval and commit | `harness-cli session approval inspect <SESSION_ID> [--task-id <TASK_ID>]` | `--json` |
| Worker lane and diff | `harness-cli session worker inspect <SESSION_ID> [--task-id <TASK_ID>]` | `--json` |
| Architecture | `harness-cli report architecture` | `--json` |
| Data flow | `harness-cli report data-flow` | `--json` |
| Storage | `harness-cli report storage` | `--json` |

Timeline and worker lane reports also accept `--payload-bytes <N>` to bound
large EventLog payloads, stdout/stderr, `git status`, and `git diff` summaries.

## Source Of Truth

The V0.4 reports are projections. They are derived from:

- PostgreSQL `harness_runtime.event_log`, the append-only runtime EventLog.
- PostgreSQL `harness_runtime.task_queue`, the runtime-owned queue/lease table.
- Static runtime-owned report definitions for architecture, data-flow, and
  storage diagrams.

Inspect/report commands must not:

- Run workers.
- Allocate worktrees.
- Capture new diffs.
- Append EventLog records.
- Change queue, lease, approval, or commit state.
- Commit or push user code.

## Manual Acceptance Path

Run acceptance against a disposable Git repository so the worker diff and commit
handoff do not touch source fixtures.

```powershell
cargo run -p harness-cli -- migrate --database-url $env:HARNESS_DATABASE_URL

$repo = Join-Path $env:TEMP "coding-agent-harness-v04-observability"
Remove-Item -Recurse -Force $repo -ErrorAction SilentlyContinue
New-Item -ItemType Directory -Path $repo | Out-Null
Set-Content -Path (Join-Path $repo "AGENTS.md") -Value "Use deterministic agent fixtures."
Set-Content -Path (Join-Path $repo "README.md") -Value "fixture repository"
git -C $repo init
git -C $repo add AGENTS.md README.md
git -C $repo -c user.name="Coding Agent Harness QA" -c user.email="harness-qa@example.invalid" commit -m "initial fixture"

$runner = Join-Path $repo "fake-success-runner.ps1"
@'
Write-Output "0123456789"
Add-Content -Path README.md -Value "changed by V0.4 observability acceptance"
exit 0
'@ | Set-Content -Path $runner

$start = cargo run -p harness-cli -- session start --repo $repo --database-url $env:HARNESS_DATABASE_URL
$sessionId = ($start | Select-String '^session_id=').Line.Split('=')[1]

$create = cargo run -p harness-cli -- session task create $sessionId `
  --database-url $env:HARNESS_DATABASE_URL `
  --task "write the V0.4 observability acceptance task"
$taskId = ($create | Select-String '^task_id=').Line.Split('=')[1]

cargo run -p harness-cli -- session task enqueue $sessionId $taskId `
  --database-url $env:HARNESS_DATABASE_URL

cargo run -p harness-cli -- session task run-next-codex-worker `
  --database-url $env:HARNESS_DATABASE_URL `
  --worker-id v04-observability-worker `
  -- powershell -NoProfile -ExecutionPolicy Bypass -File $runner
```

At this point the task should be in `pending_commit_approval`. Run the read-only
inspect/report surface:

```powershell
cargo run -p harness-cli -- session inspect $sessionId --database-url $env:HARNESS_DATABASE_URL
cargo run -p harness-cli -- session inspect $sessionId --database-url $env:HARNESS_DATABASE_URL --json

cargo run -p harness-cli -- session task inspect $sessionId $taskId --database-url $env:HARNESS_DATABASE_URL
cargo run -p harness-cli -- session task inspect $sessionId $taskId --database-url $env:HARNESS_DATABASE_URL --json

cargo run -p harness-cli -- session task queue-inspect --session-id $sessionId --database-url $env:HARNESS_DATABASE_URL
cargo run -p harness-cli -- session task queue-inspect --session-id $sessionId --database-url $env:HARNESS_DATABASE_URL --json

cargo run -p harness-cli -- session timeline $sessionId --task-id $taskId --payload-bytes 512 --database-url $env:HARNESS_DATABASE_URL
cargo run -p harness-cli -- session timeline $sessionId --task-id $taskId --payload-bytes 512 --database-url $env:HARNESS_DATABASE_URL --json

cargo run -p harness-cli -- session approval inspect $sessionId --task-id $taskId --database-url $env:HARNESS_DATABASE_URL
cargo run -p harness-cli -- session approval inspect $sessionId --task-id $taskId --database-url $env:HARNESS_DATABASE_URL --json

cargo run -p harness-cli -- session worker inspect $sessionId --task-id $taskId --payload-bytes 512 --database-url $env:HARNESS_DATABASE_URL
cargo run -p harness-cli -- session worker inspect $sessionId --task-id $taskId --payload-bytes 512 --database-url $env:HARNESS_DATABASE_URL --json

cargo run -p harness-cli -- report architecture
cargo run -p harness-cli -- report architecture --json
cargo run -p harness-cli -- report data-flow
cargo run -p harness-cli -- report data-flow --json
cargo run -p harness-cli -- report storage
cargo run -p harness-cli -- report storage --json
```

Expected signals:

- Session inspect reports `projection_kind=session_inspect_report`.
- Task inspect reports queue, lease, workspace, diff, approval, and commit slots.
- Queue inspect reports `projection_kind=task_queue_inspect_report`.
- Timeline reports `projection_kind=bounded_eventlog_timeline`.
- Approval inspect reports `projection_kind=approval_commit_inspect_report`.
- Worker inspect reports `projection_kind=worker_lane_diff_inspect_report`.
- Architecture/data-flow/storage reports include their runtime report ids.
- Re-running the inspect/report commands does not change `session show`
  `event_count` or the Task queue, lease, approval, or commit values.

## Known Limits And Non-Goals

- V0.4 is not a Web UI, TUI, iOS app, IDE panel, server, SSE stream,
  WebSocket stream, Prometheus exporter, or OpenTelemetry integration.
- Reports are point-in-time projections, not live subscriptions.
- Worker lane policy evidence is associated from the EventLog sequence because
  the current policy event payload does not carry a lane id.
- Diff evidence is associated to the latest lane in the same Task scope because
  the current diff event payload does not carry a lane id.
- Bounded payload summaries are for inspection and audit, not full artifact
  export.
- Real Codex CLI acceptance is documented separately in
  `docs/development/local-codex-cli-acceptance.md`.
