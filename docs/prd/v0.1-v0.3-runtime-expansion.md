# PRD: V0.1-V0.3 Runtime Expansion

## Problem Statement

The V0.0.1 baseline proves that the harness can run a deterministic internal
self-recovery path through CLI, PostgreSQL EventLog, Policy Gate, Tool Runtime,
bounded recovery, and pending commit approval. The next problem is that this
runtime still does not govern a real coding-agent worker, does not own the
approval-to-commit lifecycle, and does not yet model tasks as schedulable units.

The user wants the harness to become useful for real multi-agent coding work
without losing the safety and replay properties established in V0. The next
expansion must therefore connect one real local Codex CLI worker lane, make
approval and commit handoff runtime-owned, and introduce a PostgreSQL-backed task
queue with leases. This must remain a runtime-first system, not a Codex wrapper,
IDE plugin, static dashboard, or cloud scheduler.

## Solution

Build the next tranche as three vertical runtime slices.

V0.1 introduces the Local Governed Codex CLI Worker as the First Real Lane. The
harness creates or allocates a Task Worktree, starts a non-interactive One-Shot
Worker Run of the local Codex CLI under Lane-Level Governance, captures
stdout/stderr/exit status/timing, records worker lane events in EventLog,
captures the resulting diff and git status, and stops at pending commit
approval. Automated tests use deterministic fixtures or fake runners; Manual
Lane Acceptance invokes a locally installed and authenticated Codex CLI when
available, or reports an explicit skipped reason.

V0.2 introduces the Approval State Machine and Commit Handoff. Workers continue
to propose diffs only. The harness owns approval, rejection, commit execution,
commit failure reporting, and all corresponding EventLog records. The first
surface is CLI/API semantics, not Web, iOS, IDE, or GitHub review UI.

V0.3 introduces Task as a first-class runtime object and adds a single-node
PostgreSQL-backed task queue with Task Leases. Sessions remain context and
aggregation boundaries, while tasks become schedulable work units with their own
repository target, Task Worktree, worker lane, budget, status, diff, approval
outcome, and lease lifecycle.

Each version must keep a Vertical Acceptance Path: observable behavior through
the CLI, PostgreSQL EventLog, worker execution or approval state, and replayable
evidence.

   harness remains the source of truth.

   that multiple agents do not collide in my current working tree.

   opt-in, so that dangerous debugging modes are not used accidentally.
6. As the project owner, I want worker runs to be non-interactive one-shot runs,

## Implementation Decisions

- The next tranche is split into three vertical releases: V0.1 real Codex worker
  lane, V0.2 approval and commit handoff, and V0.3 task queue and worker
  scheduling.
- V0.1 implements Codex as the First Real Lane and does not implement Claude
  Code, OpenCode, Gemini CLI, Qwen Code, or multi-provider routing.
- V0.1 invokes a local Codex CLI only. Codex Cloud, remote daemon control, and
  web-linked Codex execution are out of scope.
- Codex runs are Local Governed Codex CLI Workers: external workers that produce
  evidence, not sources of truth.
- Codex runs are One-Shot Worker Runs. V0.1 does not support interactive chat,
  persistent conversation sessions, or long-running background daemons.
- Codex runs default to Task Worktrees. Running directly in the current working
  tree is a dangerous/debug path and must require explicit policy handling.
- Governance for real Codex execution is Lane-Level Governance. V0.1 controls
  worker startup, working directory, timeout, budget, environment, output bounds,
  cancellation, and result capture; it does not intercept every internal Codex
  tool call.
- Worker execution must record requested, queued or accepted, running, terminal
  status, output observation, and diff observation events.
- Worker terminal statuses must distinguish success, failure, timeout,
  cancellation, rejection, and missing executable or unavailable local setup.
- Worker usage must distinguish officially reported usage, local estimate, and
  unknown or unavailable usage.
- Automated tests must use deterministic fixture or fake runners instead of real
  Codex login state.
- Manual Lane Acceptance must attempt the real local Codex CLI when available
  and emit an explicit skipped reason when unavailable.
- V0.1 ends with reviewable output and pending commit approval. It does not
  commit or push.
- V0.2 introduces an Approval State Machine owned by the runtime.
- Approval state must distinguish pending approval, approved, rejected,
  committing, committed, and commit failed outcomes.
- Approval and rejection must be EventLog records, not UI-only state.
- Commit Handoff is owned by the harness. Workers propose diffs; the harness
  records approval and performs the commit operation.
- V0.2 may support commit, but not automatic push or remote merge.
- The first approval surface is CLI/API semantics. Web, iOS, IDE, and GitHub PR
  review surfaces are future clients.
- V0.3 introduces Task as a first-class runtime object.
- Session remains a context and aggregation boundary. Task is the schedulable
  unit.
- A Task owns or references its input, repository target, base revision, Task
  Worktree, worker lane, budget, lease state, status, diff, approval outcome,
  and commit outcome.
- V0.3 uses a single-node PostgreSQL-backed queue. It does not introduce
  Kubernetes controllers, cloud queues, remote worker fleets, or distributed
  scheduling.
- Task Lease is the first scheduling primitive. One worker can lease a task at a
  time, and expired leases can be retried or reassigned.
- Task projections must be derived from EventLog and database state owned by the
  runtime, not from client-specific state.
- Each version must preserve the V0.0.1 source-of-truth boundary: PostgreSQL
  EventLog and runtime projections remain authoritative.
- Each version must expose a Vertical Acceptance Path through CLI, PostgreSQL,
  EventLog replay, and visible task or approval state.

## Testing Decisions

- The primary test seam for the whole tranche is the CLI plus PostgreSQL EventLog
  Vertical Acceptance Path.
- Tests should assert external behavior: CLI output, recorded events, replayed
  state, task projections, worker output capture, diff capture, approval
  outcome, commit outcome, queue state, and lease transitions.
- Tests should not assert private implementation details of runner internals,
  projection internals, or Codex internal tool calls.
- V0.1 automated tests should use fake or fixture Codex runners to cover
  executable discovery, policy rejection, environment shaping, timeout,
  cancellation, stdout/stderr capture, exit code capture, usage confidence, diff
  capture, and final pending approval state.
- V0.1 manual acceptance should use a local authenticated Codex CLI when
  available and report an explicit skipped reason when not available.
- V0.1 tests should prove that current-worktree execution is not the default and
  that Task Worktree execution records the worktree identity.
- V0.2 tests should cover approve, reject with reason, commit success, commit
  failure, and replay into task state.
- V0.2 tests should prove that workers do not own commit execution.
- V0.3 tests should cover task creation, enqueue, list/show projection, lease
  acquisition, lease exclusivity, heartbeat or renewal, completion, failure,
  cancellation, timeout, and expired lease recovery.
- V0.3 tests should remain single-node and PostgreSQL-backed.
- Existing runtime, CLI, policy, tools, events, and database test seams should be
  extended before adding lower-level seams.
- The release check set remains formatting, workspace tests, workspace build,
  clippy with warnings denied, and the relevant manual acceptance path for that
  version.

## Out of Scope

- Codex Cloud execution.
- Remote daemon control.
- Interactive Codex sessions.
- Long-running external worker conversations.
- Intercepting every internal Codex tool call.
- Treating Codex session state as the harness source of truth.
- Claude Code, OpenCode, Gemini CLI, Qwen Code, or other real external lanes.
- Multi-provider model or worker routing.
- Web dashboard.
- iOS client.
- IDE plugin.
- GitHub PR review UI.
- Automatic push.
- Remote merge.
- Kubernetes controller.
- Multi-machine worker fleet.
- Production deployment, high availability, replication, and backup automation.
- Full MCP client/server support.
- Skill marketplace.
- Large-scale RAG or knowledge graph work.

## Further Notes

This PRD extends the V0.0.1 baseline. It does not replace the V0 source-of-truth
decisions. EventLog remains authoritative; clients and workers remain surfaces
or evidence producers.

The most important constraint is that the harness must not become a thin Codex
wrapper. Codex is the First Real Lane because it is the closest real worker for
the current workflow, but the durable runtime concepts are Task, Session, Task
Worktree, Lane-Level Governance, Approval State Machine, Commit Handoff, Task
Lease, and Vertical Acceptance Path.
